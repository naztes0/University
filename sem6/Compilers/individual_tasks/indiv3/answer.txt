Питання: чи буде демонструвати контейнерне середовище buildenv потрібну поведінку з коробки? 

Ні, не демонструє 
У перевіреній гілці master репзиторію buildenv взагалі відсутній Makefile, через що перша ж команда make start одразу завершиться з помилкою: No rule to make target 'start'.
Навіть якщо взяти інші гілки, де Makefile присутній, його таргет start виконує лише команди docker start та docker attach для вже існуючого контейнера. Він не створює його з нуля і не налаштовує зв'язок із файловою системою.

Через те, що відсутній повний сценарій ініціалізації контейнера- наразі в скриптпх немає логіки, яка б під час першого запуску автоматично створювала контейнер і монтувала локальне дерево файлів нашого проєкту (tc-base.plain) всередину віртуального середовища.

Для виправлення треба додати Makefile з таргетом ensure-container від якого залежить start. При виклику make start спочатку відпрацьовує ensure-container який перевіряє чи на хості взагалі існує директорія з проєктом компілятора ../tc-base.plain, за потреби підтягне образ через docker pull. Далі створює контейнер і монтує локальну директорію ../tc-base.plain всередину як /opt/tiger. Ключова частина - bind mount:../tc-base.plain:/opt/tiger. Він забезпечує спільне дерево фвйлів між хостом і контейнером. Завдяки цьому файл створений в контейнері одразу видно на хості і навпаки, що і підтверджує приклад з відео з echo "test" > nw в контейнері та cat nw на хості.
